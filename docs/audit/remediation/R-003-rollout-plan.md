# R-003 rollout plan — Default-deny project isolation (DEF-SEC-001)

**Status:** Ready for staged activation  
**Default production mode:** `PROJECT_ACCESS_ENFORCEMENT=enforce`  
**Temporary observe mode:** staging-only, time-boxed, must be removed

## 1. Route inventory completion

- Generated: `docs/audit/remediation/R-003-project-route-inventory.csv`
- Regenerated by: `node scripts/r003-scan-route-scopes.mjs`
- Static gate: `apps/backend/src/modules/project-access/route-scope.static.spec.ts`
- Every authenticated handler must have an explicit scope (`project` / `global` / `investor` / `public` / `system` / `webhook`).

## 2. Client compatibility changes

| Client | Change |
| --- | --- |
| Web | Continues sending `X-Project-Id` from active project; omits header on `/investor-portal/*` |
| Mobile | Interceptor no longer overwrites explicit `X-Project-Id` (offline sync retains txn project) |
| Mobile | Project switch clears stale selection and invalidates non-project-list caches |
| Investor portal | Path assert before requests; unauthorised selection rejected; project switch invalidates portal queries |

CORS allows `X-Project-Id`.

## 3. Staging-only enforcement mode

```bash
# Staging observation (temporary)
PROJECT_ACCESS_ENFORCEMENT=observe

# Production / default
PROJECT_ACCESS_ENFORCEMENT=enforce
```

Observe mode:

- Logs `PROJECT_ACCESS_OBSERVE` warnings and writes unauthorized-attempt audits
- Still records denials for unclassified routes / mismatches
- **Hard-denies cross-company access** even when `observe` is set (R-003B)
- May only soft-allow missing project metadata / assignment gaps in staging
- Must not ship as a permanent production default
- **Removal/expiry:** remove `observe` path after staging metadata cleanup (target ≤ 30 days post-cutover); keep production default `enforce`

## 4. Logging-only observation period

Use only if staging shows unexpected 403 spikes after client deploy:

1. Deploy clients first (web/mobile/investor).
2. Enable `observe` in staging for ≤ 72 hours.
3. Review `unauthorized_project_access_attempts` + ops logs for false positives.
4. Fix client/route classification issues.
5. Flip to `enforce` before production cutover.

## 5. Deny-mode activation

1. Confirm static scanner green in CI.
2. Confirm project-access unit + guard e2e + IDOR matrix green.
3. Set `PROJECT_ACCESS_ENFORCEMENT=enforce` (default).
4. Deploy backend, then confirm clients already sending `X-Project-Id` where required.

## 6. Monitoring for 403/404 spikes

Watch:

- Rate of HTTP 403 on `/api/v1/*`
- Inserts into `unauthorized_project_access_attempts`
- Client error reports for “Access denied” after project switch
- Offline mobile sync failures after project switch

## 7. Emergency rollback

1. **Preferred:** fix classification / client header bugs and redeploy.
2. **Temporary:** set `PROJECT_ACCESS_ENFORCEMENT=observe` for ≤ 24 hours in the affected environment only.
3. **Do not** remove route-scope decorators or restore opt-in no-op guard behaviour.
4. Open a removal ticket for any observe flag before the next release.

## 8. No permanent allow-by-default flag

There is no feature flag that restores “undecorated = allow”.  
`observe` is deny-logging only for unclassified/mismatch paths while still allowing the request; it is temporary and production defaults to `enforce`.
